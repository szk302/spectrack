# `spectrack log` コマンド テストケース一覧

## 1. 基本動作（正常系：バージョン変更履歴の抽出）

| ID | テスト項目 | 事前条件 | 実行コマンド | 期待される結果 |
| :--- | :--- | :--- | :--- | :--- |
| **L1-1** | 通常の履歴表示 | `a.md` のバージョンが過去に3回（それぞれ別コミットで）更新されている | `spectrack log docs/a.md` | 終了コード `0`。3回分のバージョン、更新日時、コミットハッシュが新しい順にリスト形式で出力されること |
| **L1-2** | 初回コミットのみのファイル | `a.md` が `1.0.0` としてコミットされた直後であり、以降の更新がない | `spectrack log docs/a.md` | 終了コード `0`。初回コミットの `1.0.0` の履歴が1行だけ表示され、正常終了すること |
| **L1-3** | バージョンのダウングレード（巻き戻し） | `a.md` のバージョンが `1.0.0` → `2.0.0` → `1.5.0` の順でコミットされている | `spectrack log docs/a.md` | 終了コード `0`。セマンティックバージョンの大小に関わらず、「変更された」という事実ベースで時系列順にすべての履歴が表示されること |

## 2. フィルタリングロジック（Filtering & Uncommitted）

一般的な `git log` と異なり、仕様書としての「意味のあるバージョン更新」だけを抽出できているかを検証します。

| ID | テスト項目 | 事前条件 | 実行コマンド | 期待される結果 |
| :--- | :--- | :--- | :--- | :--- |
| **F2-1** | 本文のみのコミットの除外（ノイズ除去） | `a.md` に対して合計5回のコミットがあるが、フロントマターの `version` が書き換わったコミットは2回だけである | `spectrack log docs/a.md` | 終了コード `0`。本文のみの修正コミット（3回分）はスキップされ、**バージョンが変更された瞬間の2コミットのみ**が抽出・表示されること |
| **F2-2** | 未コミットのバージョン更新（Working Tree） | 最後のコミットから、フロントマターの `version` を更新しているが、まだコミットしていない | `spectrack log docs/a.md` | 終了コード `0`。コミット履歴の先頭（最新）に `(Working Tree)` 等のステータスで未コミットの新しいバージョンが含まれて表示されること |
| **F2-3** | 未コミットの本文更新（バージョン維持） | 本文は編集している（Git差分あり）が、`version` は最後のコミットと同じである | `spectrack log docs/a.md` | 終了コード `0`。バージョン自体は変わっていないため、Working Tree のエントリは新規追加されず、最新のコミットが先頭として表示されること |

## 3. 履歴の追跡と境界値（Tracking & Boundary）

| ID | テスト項目 | 事前条件 | 実行コマンド | 期待される結果 |
| :--- | :--- | :--- | :--- | :--- |
| **T3-1** | ファイルのリネーム追跡（`--follow` 相当） | `old.md` として運用しバージョンを2回更新した後、`new.md` にリネーム（移動）してさらに更新した | `spectrack log docs/new.md` | 終了コード `0`。リネーム前の `old.md` 時代のバージョン更新履歴も途切れずに遡って追跡・表示されること |
| **T3-2** | ネストされたバージョンパスの解決 | `.yml` ファイルで `x-st-version-path: info.version` と設定されている | `spectrack log docs/api.yml` | 終了コード `0`。現在のファイルだけでなく、過去の全コミットにおいても `info.version` のネストされた値を正確にパースして抽出できること |

## 4. Git環境依存とエラーハンドリング（Git Dependency）

このコマンドは「Gitコミット履歴を正として動作する」ため、Gitがない場合は明確にエラーにする必要があります。

| ID | テスト項目 | 事前条件 | 実行コマンド | 期待される結果 |
| :--- | :--- | :--- | :--- | :--- |
| **G4-1** | Git未初期化ディレクトリでの実行 | プロジェクトが `git init` されていない | `spectrack log docs/a.md` | 終了コード `1`。生のGitエラーを隠蔽し、「ERROR: Git リポジトリが初期化されていません」の共通エラーを出力して中断されること |
| **G4-2** | Gitコミットゼロの状態 | `git init` 済みだが、まだ1つもコミットが存在しない | `spectrack log docs/a.md` | 終了コード `1`。「ERROR: 少なくとも1つのコミットが必要です」のエラーを出力して中断されること |

## 5. 異常系・履歴内のパースエラー耐性（History Errors & Resilience）

過去のコミットをパースする際、過去の自分が「壊れたYAML」をコミットしてしまっていた場合にツールがどう振る舞うかのテストです。

| ID | テスト項目 | 事前条件 | 実行コマンド | 期待される結果 |
| :--- | :--- | :--- | :--- | :--- |
| **E5-1** | 対象ファイルが存在しない | 存在しないファイルパスを指定する | `spectrack log docs/not_found.md` | 終了コード `1`。「ファイルが見つかりません」のエラーが出力されること |
| **E5-2** | 対象ファイルが未初期化 | `a.md` の現在の Working Tree にフロントマターがない | `spectrack log docs/a.md` | 終了コード `1`。「対象ファイルが初期化されていません」等のエラーとなり中断されること |
| **E5-3** | 過去のコミットでフロントマターが破損 | 現在のファイルは正常だが、3つ前のコミットでYAMLが完全に壊れている状態（構文エラー）が保存されている | `spectrack log docs/a.md` | 終了コード `0`。過去ファイルのパース時にクラッシュせず、壊れたコミットはスキップするか `[バージョン取得失敗]` 等の表示にフォールバックして、全体のタイムライン出力は完了すること |
| **E5-4** | 過去のコミットでキーが欠損 | 過去のコミットの中に、`x-st-version-path` が示すキー自体が存在しない状態がある | `spectrack log docs/a.md` | 終了コード `0`。これもクラッシュせず、「バージョン不明」として安全にハンドリングされること |
