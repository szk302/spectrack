# `spectrack verify` コマンド テストケース一覧

## 1. 基本動作（正常系・完全整合 / Exit Code: 0）

すべてのファイルが正しく設定され、整合性が取れている状態のテストです。

| ID | テスト項目 | 事前条件 | 実行コマンド | 期待される結果 |
| :--- | :--- | :--- | :--- | :--- |
| **V1-1** | プロジェクト全体の正常検証 | プロジェクト内の全ファイルのフロントマター、ID、依存先パス、バージョンがすべて正常 | `spectrack verify` | 終了コード `0`。すべてのチェック項目が `OK` となり正常終了すること |
| **V1-2** | 追跡対象ファイルが0件 | プロジェクト内にフロントマターを持つ対象拡張子ファイルが1つもない | `spectrack verify` | 終了コード `0`。「検証対象のドキュメントがありません」等のメッセージで正常終了すること |
| **V1-3** | 除外設定の適用 | `.spectrackignore` 配下のファイルにエラーがある | `spectrack verify` | 終了コード `0`。除外対象はスキャンされないため正常終了すること |
| **V1-4** | Git未初期化ディレクトリ | プロジェクトが `git init` されていない | `spectrack verify` | 終了コード `0`。Working Treeのファイル情報のみで検証が完結するため正常終了すること |

## 2. 循環依存の検証（Warnings / Options）

| ID | テスト項目 | 事前条件 | 実行コマンド | 期待される結果 |
| :--- | :--- | :--- | :--- | :--- |
| **C2-1** | 単純な循環依存の検知（デフォルト） | `a.md` が `b.md` に依存し、`b.md` も `a.md` に依存 | `spectrack verify` | **終了コード `2`**。循環依存の警告（`⚠️ 警告`）が表示され、該当するループパスが出力されること |
| **C2-2** | 複雑な循環依存の検知（デフォルト） | `a.md` -> `b.md` -> `c.md` -> `a.md` という3ファイル以上のループが存在 | `spectrack verify` | **終了コード `2`**。スタックオーバーフローにならずにループパスを特定し、警告として出力されること |
| **C2-3** | **`--allow-cycles` の適用（全許容）** | **`C2-1`（単純な循環）および `C2-2`（複雑な循環）の両方の状態がプロジェクト内に混在している** | `spectrack verify --allow-cycles` | **終了コード `0`**。パターンの複雑さに関わらず、**すべての循環依存が正常な参照として許容され**、エラーや警告が一切出力されずに正常終了すること |
| **C2-4** | 自己参照（自己への依存） | `a.md` の依存リストに `a.md` 自身のIDが含まれている | `spectrack verify` | **終了コード `1` または `2`**。自己参照は無意味なループのため、個別の警告または致命的エラーになること |

## 3. 致命的エラーの検証（Fatal Errors / Exit Code: 1）

CIのパイプラインを止めるべき、プロジェクトの整合性を破壊するエラーのテストです。

| ID | テスト項目 | 事前条件 | 実行コマンド | 期待される結果 |
| :--- | :--- | :--- | :--- | :--- |
| **E3-1** | IDの重複（一意性違反） | `a.md` と `b.md` の `x-st-id` の値が完全に一致している | `spectrack verify` | **終了コード `1`**。「ID重複検出」のエラーが出力され検証失敗となること |
| **E3-2** | 依存先の解決失敗（リンク切れ） | `a.md` が依存している ID がプロジェクト内に存在しない | `spectrack verify` | **終了コード `1`**。「参照先解決: エラー」が出力され検証失敗となること |
| **E3-3** | バージョン情報の欠損 | `a.md` に `x-st-id` はあるが、`version` キーが欠損している | `spectrack verify` | **終了コード `1`**。バージョン取得不能エラーが出力され検証失敗となること |
| **E3-4** | フロントマターの構文エラー | 先頭のYAMLがインデント不正などでパース不能に壊れている | `spectrack verify` | **終了コード `1`**。クラッシュせず「フロントマター形式が不正です」と出力されること |
| **E3-5** | バイナリの混入 | `.md` 拡張子のまま中身がバイナリデータになっている | `spectrack verify` | **終了コード `1`**。文字化けせず、パースエラーとして安全に捕捉されること |
| **E3-6** | IDキー自体の欠損 | フロントマターはあるが、`x-st-id` キーが空文字または存在しない | `spectrack verify` | **終了コード `1`**。「IDが未設定または不正です」として検証失敗となること |

## 4. セマンティックバージョニングの検証（Warnings / Exit Code: 2）

| ID | テスト項目 | 事前条件 | 実行コマンド | 期待される結果 |
| :--- | :--- | :--- | :--- | :--- |
| **W4-1** | SemVer非準拠のバージョン | `a.md` のバージョンが `1.0` や `1.0.0-alpha` 等である | `spectrack verify` | **終了コード `2`**。「有効なセマンティックバージョンではありません」という警告が出力されること |
| **W4-2** | バージョンの型不正 | `a.md` のバージョン値が数値や配列（`[1,0,0]`）になっている | `spectrack verify` | **終了コード `1` または `2`**。文字列としてパースできないため、不正な型として扱われること |
